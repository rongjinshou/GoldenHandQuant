# round-18 诊断：平台"3 小时到 B11 自动关闭"根因 = 主 agent 上下文耗尽

## 一、平台现象

平台工程反馈：任务运行 3 小时、跑到 **B11 agent 就自动关闭**，无输出结果；怀疑上下文过大
（平台上下文上限 **200K**）。

## 二、诊断方法与结论（实测，非推断）

数据源：opencode 自身的会话库 `~/.local/share/opencode/opencode.db`（`message.tokens` 记录
每轮真实 input/cache token）+ round-17 三判运行日志。

**结论：不是超时，不是崩溃，是主 agent 上下文线性爆炸后被运行时掐断。**

主会话（round-17 judge1，03:06→05:07）上下文增长曲线：

```
03:06=7K → 03:25=27K → 03:32=81K → 03:42=139K → 03:59=176K
→ 04:11=200K ←【平台在此关闭】→ 04:21=253K → 04:30=302K → 04:45=361K → 05:06=442K
```

**04:11 越过 200K，对应第 10~12 批——与平台"3 小时到 B11 关闭"精确吻合。** 本地能跑完
19 批，仅因为 mimo-v2.5 的上下文窗口远大于平台的 200K 配额；平台配额下必然在 B11 前后中断，
后半程批次全部丢失 → 无输出结果。

## 三、构成拆解：84% 来自一件纯粹的浪费

主 agent 会话的 part 统计：

| 工具 | 次数 | 体量 | 占比 |
|---|---|---|---|
| **read** | **40** | **1,840,133 字节 ≈ 460K token** | **84%** |
| todowrite | 20 | ~31K token | 6% |
| edit（全部是写 result/output.md） | 19 | ~15K token | 3% |
| **task（派遣 subagent）** | 22 | **~14K token** | 3% |
| bash | 52 | ~12K token | 2% |

40 次 read 里 13 次是**整个卡片文件**：cart.md 102KB、user.md 95KB、inventory.md 83KB、
S3-audit.md 68KB、common.md 64KB、payment.md 59KB ……

**关键事实：主 agent 读完这些卡片后什么也没做**——真正动手的是 subagent，它必须自己再读
一遍同一个文件。主 agent 那一遍是 100% 浪费，且是致命的浪费。

**对照组证明 harness 本身是省的**：派遣(task) 仅 14K、bash 52 次仅 12K、
`ratchet.sh verify` 单次 stdout 只有 **188 字节**。问题不在 harness 输出设计，
纯粹出在 read 卡片这一项。

## 四、根因：INSTRUCTION 措辞的主语混乱

第③步 a 原文：

> a. 按批次表**打开该批对应的卡片文件**（…），逐卡修改 code/ 下的源文件
>    —— 必须派 subagent 执行本批，…

前半句读起来就是命令主 agent 自己打开卡片，后半句才提派遣。弱模型（mimo-v2.5）照字面顺序
执行：先 read 一遍卡片（500 行/次），再派 subagent。

讽刺的是，同一步下方本来就写着「派 subagent 不是可选优化：它把整批长卡片隔离在子上下文里，
主上下文每批只留一行结果，这是 19 批全部跑完的前提」——**设计意图完全正确，只是被上一句的
措辞废掉了**。隔离机制只在"主 agent 自己不读"时才成立，而这个前提从没被写出来。

## 五、修复（commit f669aa9）

1. **第③步开头加绝对禁令**：主 agent 永不 read `work/bugs/` 下任何卡片文件，只读 README
   批次表；写明后果（第 10~12 批顶爆、会话被掐断、后半程全丢）与每批 ~3K token 的预算，
   并给出自查点"若发现自己正准备 read 卡片——停下，改为派遣"。
2. **a 步改写**：「查表即可，不要打开该文件」，由 subagent 打开卡片。
3. **补上隔离机制的前提说明**：subagent 隔离只在"你自己不读卡片"时才成立；先读再派 =
   隔离形同虚设 + 那一遍读毫无用处。
4. **SKILL.md**：卡片**一次读完、只读一遍**——read 默认 limit=500 而卡片有 1000~2100 行；
   实测 subagent 把 promotion.md 整体读了两遍（88,668 + 84,363 字节），白烧一倍上下文。
5. 经 `install-agent.sh` 重新生成 `.opencode` 三份预注册副本，四份保持同源。

**预期：主上下文 446K → ~100K**，安全落在 200K 内。

## 六、耗时的真相：Maven 完全不是瓶颈

实测单次耗时（judge1 树，离线仓库）：

| 环节 | 耗时 | 占比 |
|---|---|---|
| `mvn install -DskipTests` | **2.7 秒** | 1% |
| `mvn test`（24 例黑盒） | **25.1 秒** | 10% |
| **LLM 推理** | **~236 秒** | **89%** |

每批 4.4 分钟里构建只占 28 秒。**"减少耗时"= 减少 token 处理量，合并批次/优化构建都无效。**
且上下文膨胀是复合灾难：attention 是 O(n²)，上下文越大每轮越慢——这解释了平台每批 16 分钟
vs 本地 4.4 分钟的差距（平台主 agent 长期在 200K 附近推理）。修上下文同时就是修耗时。

批次耗时与卡片体量强相关（本地 round-17 judge1）：

| 批次 | 耗时 | 卡片体量 |
|---|---|---|
| B05 promotion | 8.2 分钟 | 117KB / 19 卡 |
| B03 order §A | 6.7 分钟 | 102KB / 31 卡 |
| B01 S1-quick-wins | 1.6 分钟 | 3.9KB |

subagent 单批存活 6~8 分钟 ≈ 整批耗时——**耗时几乎全在 subagent 干活**，主 agent 决策占比小。
含义：修主 agent 上下文能救回"跑不完"，但未必大幅压缩总耗时。

## 七、待决问题（需向平台确认，本地无法推断）

**平台那 3 小时是"上下文爆掉"还是"硬超时"？** 两者解法完全不同：

- 若是**上下文爆掉**（当前证据强烈支持）：f669aa9 已解决，平台应能跑完 19 批。
- 若还有**3 小时硬超时**：本地 19 批需 92 分钟，平台约 5 小时（每批 16 分钟）——
  修上下文后主 agent 变快，但 subagent 耗时不变，仍可能超。届时唯一有效手段是**卡片瘦身**
  （减少 subagent 要读的字数），代价是降低弱模型理解力，需实测权衡。

## 八、subagent 侧的两个浪费（已一并修复）

实测 63 个 subagent 会话峰值：最大 **159,389**（80% of 200K）、中位 **74,316**、超 200K **0 个**。
即"目前没爆，但最危险的 3 个只剩 20% 余量"。注意严重性分级：**subagent 爆掉只丢一批**
（verify 判回滚 → 重试一次 → 跳过），主 agent 爆掉是后半程全丢——差一个量级。

挖出的两个原因都不是"卡片天生太长"：

### ① subagent 重复读同一卡片（1.6~1.7 倍）

| 文件 | 实际体量 | subagent 读入 | 倍数 |
|---|---|---|---|
| promotion.md | 117KB | **197,640 字节** | 1.7× |
| order.md | 102KB | **160,953 字节** | 1.6× |

根因：read 工具默认 `limit=500` 行，而卡片有 1000~2100 行 —— 分段读时读重了。
修复：SKILL.md 明确"一次读完（limit 给 3000）、只读一遍"。

### ② PROMO-16 指针卡强制跨文件读（本轮最大单点浪费）

上下文最高的三个 subagent（159K / 158K / 148K）**全部是 B05 promotion 批**，且都读了
order.md（47~59KB）。追查发现是设计缺陷：`PROMO-16` 的改法写着
「**打开 `work/bugs/order.md`**，先定位 `### ORD-A17` 逐字照做，再定位 `### ORD-A21`」
—— **为读 2 张卡（合计 10.7KB），subagent 被迫整体载入 72.7KB / 31 张卡的 order.md，
85% 是无效上下文。**

修复（commit c90382f，严格正交）：
- PROMO-16 由指针卡改为**自包含卡**，ORD-A17/A21 **原样内嵌**（校验逐字相同：5511/5239 字符），
  标题降为 `####` 子节以保持 promotion.md 的 `###` 卡片计数不变（19 → 19）；
  改法明确"不要去打开 order.md"。
- order.md 原位留**墓碑**：保留 `###` 卡号（loyalty.md 的 "ORD-A17 同款边界" / "ORD-A21 所加"
  等**代码锚点**引用不失效）+「B03 直接跳过、完整内容见 PROMO-16、产物断言登记为 B05」说明
  —— B03/B04 的跳过行为与原来完全一致，卡数 31 → 31。
- `artifacts.tsv` **零改动**（B05 的两条 `releasePromotions` 断言原封，核验同样的产物）。
- 全库扫描确认：**再无任何卡片要求打开其他卡片文件**。

体量：order.md 72.7K → 62.6K 字符（B03/B04 顺带受益）；promotion.md 89.5K → 99.6K；
B05 净省约 61KB ≈ 15K token。两项合计预计把 subagent 峰值从 159K 压到 ~124K（余量 20% → 38%）。

## 九、一个本地环境陷阱（与包无关，但复刻者必看）

复验轮首跑**卡死 67 分钟零输出**：本机 `~/.local/share/opencode/opencode.db` 已累积到
**729MB**（15+ 轮仿真），opencode 启动后卡在 init 之后 `do_epoll_wait` 死等 I/O、CPU 0.7%、
一行活没干。这同时解释了 round-16 的 `database is locked`（judge2 严格同时启动即崩）——
**库越大，启动事务越久、锁窗口越宽**。

平台是全新环境、db 为空，不受影响；包内也无可防御点（崩/挂都发生在读 INSTRUCTION 之前）。
本地仿真的正确做法：每轮用独立 `XDG_DATA_HOME`（round-19 起已如此），原库归档留证。
