# 评测模拟复盘 — 第十一/十二轮：真实平台反馈驱动的适配 + JDK21 终版认证

## 背景：第一次真实平台提交的三份关键情报

首次平台评测（final 61 = accuracy 89% / stability 33%，三 executor 复现制）与随后确认的
评测机环境（Ubuntu 24.04 / **完整 BiSheng JDK 21** / Maven 3.9.11 / 只读作品目录 + 独立
工作目录 / 用例集可能为全量 72 例）暴露了四类此前模拟覆盖不到的问题：
executor_3 因工作目录假设失败（artifact 无效）、executor_2 时间耗尽且续跑漏批、
PROMO-6 卡漏写 pom 依赖步骤造成每个环境的 B05 都在"违规 vs 弃卡"两难中掷骰、
以及协议硬编码 24 在全量用例集下会让补救循环永不触发。

## 第十一轮：修复包（不跑模拟，全部改完统一验证）

- **第⓪步**：显式适配"只读作品目录 + 可写工作目录"布局（可写性检查 → 整包复制 → 绝不写原目录）。
- **环境快速失败**：评测机预装 JDK 21（release 17 交叉编译，17~21 均可）；删除全部便携安装/
  `~/tools` 探测/兜底——缺环境=如实记录并停止，绝不自行下载或去不存在的目录找。
  三脚本守卫升级为 java+javac+mvn 存在且主版本 ≥17（javac 检查防 JRE-only 穿透）。
- **用例数无关化**：ratchet 每条结论行携带 `total=`；补救循环触发条件 best<total；全部协议
  文本去硬编码 24。
- **逐用例回归门**：verify 记录失败用例集合，任何"此前通过的用例本批变红"即回滚（即使总数
  持平）——封死"修 A 坏 B"的等额交换（stability 杀手）。注入实测：破坏注册状态 →
  精确点名 pub001/pub105 新增失败并回滚恢复。
- **续跑升级**：进度真相 = 记录 ∪ 全批 check-batch 扫描（防"verify 完成、记录未写"间隙被杀
  后的漏批/重复）。
- **PROMO-6 补 pom 步骤**（`spring-security-core`，明确 starter 禁用原因）+ B05 pom 断言。
- **弃项复活（P1-9）**：券/秒杀取消释放（PROMO-14/15 + ORD-A17 接线卡，经 **PROMO-16 指针卡**
  随 B05 落地以维持"批=卡片文件"派遣不变式）、物流回调冗余轨迹去重（LOGI-10）——均在参考
  实现上实修并过 24/24 门禁（JDK17 与 JDK21 双环境）；券适用类目校验与商品详情 skuList
  经查证依据不足，有据放弃。artifacts.tsv 新增 5 条行为载重断言。
- **JDK21 体检**：参考实现在 Temurin 21 下 24/24 零翻转——协议与代码对 JDK 档位不敏感，
  此前公司机器的曲线异常源于该机工具链（JRE-only），非代码问题。
