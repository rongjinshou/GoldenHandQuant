# 评测模拟复盘 — 第七轮：混沌测试（中途 SIGKILL + 重启续跑）

第七轮人为制造最恶劣的中断（`SIGKILL -9`，进程无任何清理机会），验证第六轮后新增的
INSTRUCTION「重启/续跑场景」指引。

## 实验设计与结果

- **阶段一**：正常起跑（全新工作目录、全局注册清空），跑到 **B08 固化**（第 8 次 verify，
  19:30:10）后 17 秒，SIGKILL 击杀 opencode 进程（19:30:27）。
- **阶段二**：30 秒后同目录重启，观察续跑行为：
  1. agent 在执行第①步复制**之前**先跑 `ratchet.sh status`，发现 golden 存在、best=24；
  2. 原话："这是续跑场景……按照指令跳过源码复制，先 verify 对齐"→ `OK_NO_CHANGES pass=24`，
     **零损失、零重建**（对比第六轮意外断网时旧文本的"重做→回滚→重试"约 40 分钟绕路）；
  3. 读 `result/output.md` 记录定位续跑点 B09；agent 曾短暂误判"代码已全部修复"，
     **check-batch 当场纠偏**（B09 断言 3/3 缺失）——它随即自发把 B09-B19 全部扫了一遍
     产物核验，重建出完整进度真相（B01-B08 已固化 / B09-B19 全部待做）后正常派遣续跑；
  4. B09-B19 逐批照常固化，收尾终验 + `STATUS: DONE`。
- **总账**：两阶段合计 19 次 verify **零回滚**、19/19 批产物核验全绿、24/24；续跑特有开销
  （status + 空 verify + 记录对账 + 全批产物扫描）约 5 分钟。

## 结论

1. 「重启/续跑指引」按字面被弱档模型正确执行，SIGKILL 级中断的恢复成本从"半轮构建绕路"
   降到分钟级、且不产生任何回滚；
2. `artifacts.tsv` 产物清单在续跑场景自然兼任**进度 oracle**——不依赖任何上一轮口供即可
   机械重建"哪些批真正落地"，与 result/output.md 记录互为佐证；
3. 阶段一还顺带复验了 OK_NO_CHANGES 防呆：B06 首个 subagent 空手而归，主 agent 按协议判
   "本批不算已执行"并重派成功。
