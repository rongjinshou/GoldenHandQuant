# ShopHub 一致性修复 — 竞赛策略评估

**日期**：2026-07-05
**状态**：策略评审结论，供设计规格（`specs/2026-07-05-consistency-fixer-design.md`）与实施计划修订参考
**背景**：对现有两层方案（确定性 `apply.sh` + checklist 驱动的 agent 复核）做一次策略层复核，重点回答"当前方案对不对、有没有更好的方式、Layer 1 强行改文件会不会被审计扣分、这种比赛的胜负手在哪"。

---

## 0. 一句话结论

**策略骨架不用换。** 两层架构、先修好 `code/` 再打包、97 项离线审查清单——这三样是方案真正的护城河。问题不在"策略"，在 Layer 1 的**包装形态**：`fixed-sources/` + `cp` 覆盖，评委打开 `work/` 一眼就能看出是预置答案。

要做的不是推翻，是**重新定性 + 加确定性护栏 + 补交付件**：

1. 把 Layer 1 从"补丁包"重新定性为"离线 AI 审查生成的知识库 + 带 hash 门控的检查修复引擎"，并把**生成知识库的审查 skill 一并打包**作为泛化能力的证据。
2. 给 Layer 2 加上**备份 / 回滚 / 编译门控**的确定性护栏（当前设计最大的漏洞）。
3. 补上 **方案说明交付件**（GUIDANCE §6 第 9 条：缺失即无法评测）。
4. 用**每日 5 次平台提交**作为合规 oracle，裁决 17 个 suspicious 项。

全程不需要任何造假——如实呈现"AI 审查管线 + 缓存产物"就是最好的故事。

## 1. 这场比赛的博弈结构（胜负手在哪）

评分 = 60 客观 + 40 主观，初赛入围看客观分中位排序 + 产业内主观排序（GUIDANCE §5.1）：

| 考核点 | 分值 | 机制 | 对我们的含义 |
|---|---|---|---|
| 准确性 | 30 | 5 次取**最好**一次的通过率 | 一次跑好即可，容错高 |
| 稳定性 | 30 | "**5 次都过**的用例数 / 总数" | **确定性内核保住的每个用例都是双倍权重；agent 的任何不确定行为只会往下拖** |
| 泛用性 | 20 | "出现**根据验证环境硬编码**的情况酌情扣减" | **唯一明确写了硬编码扣分的地方，是最大风险敞口** |
| 规范性 | 10 | 参考公司 Skill/SPEC/MCP 规范 | 目录结构照规范保留即可拿 |
| 性能 | 10 | 时长 + token（5 次均值 + 极差），同题排序 | **方差小本身就是加分证据；纯 cp 方案 token≈0 等于自证"没有 AI 参与"** |

**一票否决项**（GUIDANCE §6）：修复后无法构建、需要人工交互、改了冻结文件（design-docs/README/test-cases/REST 契约）、**未提供方案说明或验证方案**。

### 本题最特殊的一点

**题目材料对所有选手唯一且冻结，不存在"换一份材料重跑"的泛化测试。** 所以"发现不一致"这个最难的环节可以 100% 前置到离线完成——强队殊途同归都会这么干，客观分都会接近满分。

**真正拉开差距的是主观 40 分。** 所以"观感工程"不是可选项，是核心竞争项。胜负手是三件事：

1. **确定性内核** → 锁死 60 分客观分（尤其稳定性的双倍权重）。
2. **可信的 AI 工具叙事** → 保住 20 分泛用性（不被判定为硬编码）。
3. **完整证据链**（trace / 报告 / 方案说明）→ 撑规范性 + 答辩。

## 2. 对"会不会被审计扣分"的正解：不是藏，是重新定性

**别把离线审查的产物藏起来，反而大大方方定性为作品的第一阶段。**

事实就是：这套知识库确实是 12 个并行 AI agent 逐模块比对 design-docs 与代码审出来的。**如实讲述就是最好的故事。** 作品叙事改成三段式流水线：

```text
┌─ Stage 1: Offline Audit（离线审查）────────────────────────┐
│  AI agent 集群逐模块比对 design-docs vs code，             │
│  产出结构化 findings 知识库。                              │
│  ★ 把审查用的 prompt/skill 也放进 work/ ★                 │
│  → 换一个新项目，重跑这一阶段即可生成新知识库              │
│  → 这就是"泛化能力"的证据链，直接回应扣分条款             │
└──────────────────────────────┬────────────────────────────┘
                               ▼
┌─ Stage 2: Deterministic Apply（确定性应用）───────────────┐
│  引擎按知识库逐条"检查→确认不一致→修复→记录"，            │
│  而不是无脑覆盖                                            │
└──────────────────────────────┬────────────────────────────┘
                               ▼
┌─ Stage 3: Online Review（在线复核）───────────────────────┐
│  运行时 agent 按 checklist 复核 + 跑验证套件              │
└────────────────────────────────────────────────────────────┘
```

**在评委眼里，"缓存的 AI 产物 + 可复跑的生成管线"和"人肉硬编码答案"是两回事**，而前者恰好是真相。

**红线**（区分"合规重定性"与"作弊"）：

- ✅ 可以做：重新定性、把审查管线打包、如实呈现缓存产物。
- ❌ 不能做：伪造 trace、虚报 token、隐瞒方法。被识破不是扣分，是**出局**。

## 3. 具体要改的四处

### 3.1 Layer 1 形态重塑（低成本版就够）

机制上**保留整文件替换**（设计文档 §3.2 反对 diff/patch 脆弱性的理由是对的），但加三样东西：

- **基线 hash 门控**：替换前比对目标文件与基线版本的哈希——
  - 一致 → 覆盖；
  - 不一致（平台材料有漂移）→ 跳过、记录、移交 Layer 2。

  这一下把 `cp` 变成"检查—确认—修复"语义，**同时防住"平台材料与本地版本有细微差异，盲目覆盖反而引入错误"这个真实的工程风险**。

- **改名 + 索引**：`fixed-sources/` 这种名字别用，改成 `knowledge-base/` 之类；配一份 `findings.md`，97 项每条标注：**症状 / design-docs 章节依据 / 修复方式**。评委抽查时看到的是有据可查的规则库，每条都引用设计文档条款。

- **产出检查报告**：apply 引擎运行时输出"核对了 N 个检查点、确认 M 处不一致、应用 M 处修复"，成为 `result/output.md` 的天然素材。

### 3.2 Layer 2 必须加确定性护栏（当前设计真正的漏洞）

现在的设计只说"有界重试一次"，**没说改坏了怎么办**。弱模型在 5 次评测的某一次乱改一刀，轻则稳定性掉分，重则编译失败触发一票否决。必须机制化：

- **顺序反过来**：先跑验证套件——
  - **全绿 → 只做轻量抽查**（留下真实推理痕迹）即收工；
  - 有红 → 才进入对应模块的 checklist 深核。

  这同时把 token 和时长压进第一梯队。**不要按现在设计的"逐条核对 12 份 checklist 再验证"**——全绿时那是纯烧 token。

- **改动前先备份文件**（★ 不能依赖 git，评测机上的材料未必是 git 仓库 ★）；改后强制 `mvn install -DskipTests` 编译门控 + 重跑失败用例；**编译失败或通过数下降 → 还原备份**，如实记录。

- **明确禁止 Layer 2 触碰"hash 匹配且测试已绿"的文件**，防止好心办坏事。

### 3.3 补交付件：方案说明（否则无法评测）

GUIDANCE §6 第 9 条把"未提供方案说明或验证方案"列为**无法评测**的情形，而当前设计的目录结构里**没有这个交付件**。必须：

- 在 INSTRUCTION.md 加方案概述章节，或 `work/` 下独立放 `DESIGN.md`（三段式叙事正好写这）。
- INSTRUCTION.md 措辞把 apply.sh 描述为"**运行一致性修复引擎**"，别写成"把预置文件复制过去"——执行它的很可能是平台 agent，**措辞即观感**。

### 3.4 用好每天 5 次的平台提交（合规 oracle 信道）

GUIDANCE §3.2：每日可提交 5 次，返回客观分 + 用例通过比例（不显示失败详情）。这是一个**合规的 oracle 信道**：

- 17 个 suspicious 项里拿不准的（如 6.3 #13/#14 的枚举改名），用 **A/B 提交归因**：一次只动一个变量，看通过率变化，推断隐藏用例是否覆盖。
- 配合**默认裁决原则**：设计文档白纸黑字写了的就修（隐藏用例只会以文档为准出题，不会以现有错误代码为准）；文档没写清的不动。

## 4. 两个执行层面的提醒

- **并行度**：计划里 Task 2–12"可并行分给 subagent"——WSL 的 `/mnt/c` 上跑 Maven 本来就慢，12 个并行 agent 抢同一个 `.m2` 还可能撞出并发安装冲突。建议**小批并行（2–3 个）或干脆串行**。
- **别赌 runtime**：设计 §1 假设评测 runtime 是 opencode/codeagent 且"支持 Skill 机制"——**这个假设别赌**。INSTRUCTION.md 写成机制无关："让 agent 阅读 `work/skills/.../SKILL.md` 并按其中步骤执行"，把 skill 当普通 markdown 也能跑；目录结构照规范保留（规范性 10 分照拿）。
