### T4 — 候选评估（**待定，未采纳**）：专项测试引导 agent 修绿（用户提议）
用户抛出一个想法：既然已挖出 bug、能用 cp 修，为何不写专项测试引导平台 agent 修绿？**此为待评估候选，尚未决策**（上一版我误写成"采纳"并回填蓝图，已撤回——提议 ≠ 决策，应先挑毛病、由用户拍板）。

评估——**问题多于收益，不作为核心机制**：
1. **不直接得分**：平台只按冻结的 `test-cases/`（公开+隐藏）计分，我们的专项测试不计分，只有"引导+叙事"的间接价值。
2. **最需要它时最不可靠**：只在 cp 被跳过（材料漂移）时才用得上；但漂移时目标文件≠基线，我们基于基线写的修复配方同样可能贴不上——兜底场景恰是它最靠不住的场景。
3. **给弱模型增加被带偏的面**：专项测试若过拟合（断言只有我们那版 fix 才产生的具体值），弱模型为修绿可能 hack 代码，反而过不了隐藏用例。
4. **放哪都有副作用**：注入 `code/` 模块测试目录 → 修复前是红的 → `mvn test` 失败、可能干扰平台评测流程；放 `work/` 独立工程 → 又多一套要维护、要防飞的东西。

**建议（非决策）**：
- 每个模块任务本就写 TDD 测试验证自己的修复——这个照做，是我们的质量保证，与"引导 agent"无关。
- 把这些 TDD 测试**打包成显式的、喂给 agent 的专项套件**只是锦上添花，边际收益递减，**暂不锁定**。核心仍是：确定性 cp 保证落地 + Stage 3 先验证（冻结公开黑盒 + 我们的模块测试）。
- 到打包阶段（Task 14-15）再由用户拍板是否收拢成套件。**在此之前不回填 design 蓝图。**

## T10 — 全系统集成验证（Task 13）：从 24 全 ERROR 到 24/24 全绿

12 模块单独修好合并后跑全量黑盒，先是 **24 个用例全部 ERROR**（Spring 上下文启动失败），逐层定位并修复 8 处**只在整机集成才显形**的跨模块缺陷（单模块审查看不到）：

1. **bean 名冲突**：logistics 与 loyalty 各有一个简单名 `OrderPaidEventListener` → `ConflictingBeanDefinitionException` → 上下文起不来。→ 显式模块限定 bean 名。
2. **秒杀事务毒化**：`validateSeckill`(`readOnly`) 对非秒杀 SKU 抛异常被吞，但已把共享事务标记 rollback-only → **每笔下单** commit 时 500，连锁 8 个用例。→ `noRollbackFor=ResourceNotFoundException`。
3. **AFTER_COMMIT 不落库**：物流自动建发货单在 AFTER_COMMIT 无存活事务，`save()` 不 flush、id 为 null、从不入库。→ 监听器 `REQUIRES_NEW`。
4. **pick NPE**：controller 传 null pickerId，`pick` 里 `toString()` NPE 500，断掉履约链。→ 判空。
5. **签收不联动订单**：`ShipmentDeliveredEvent` 无人监听，订单停在 PAID，评价被拒。→ 新增 order 监听器链式推到 DELIVERED（harness 的 `OrderLogisticsStatusUpdater` 是 no-op，中间态永不落订单，故设计就是靠该事件）。
6. **支付后库存不扣减**：`OrderPaymentEventHandler` 是死代码，无人监听 `PaymentSucceededEvent`。→ 新增 inventory 监听调幂等 `deductAfterPayment`（附录D §3）。
7. **verifyPurchase 排序字段不存在**：按 `deliveredAt`（Order 无此列）排序会 500。→ 改 `createdAt`。
8. **orderNo 同毫秒碰撞加固**：批量下单两单同毫秒 → orderNo 唯一约束冲突。→ 单调序列。

诊断法：读 surefire 报告 + 应用日志堆栈，`24 ERROR(上下文)→下单 500(rollback-only)→连锁`逐层剥。每修一处即重跑黑盒。**结果：公开黑盒 24/24，连续 6+ 次 0 fail 0 error 稳定。**
