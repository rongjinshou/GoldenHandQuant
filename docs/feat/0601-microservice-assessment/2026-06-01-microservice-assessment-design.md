# GoldenHandQuant 微服务拆分评估设计文档

> **版本**: v1.0 | **创建日期**: 2026-06-01 | **文档类型**: 评估设计 (Assessment Design)

## 1. 背景与目标

### 1.1 现状概述

GoldenHandQuant 当前采用 **基于 DDD 的单体架构 (Monolithic Architecture)**，所有子域（account、trade、market、strategy、risk、portfolio、notification、backtest）运行在同一进程中，通过 Python 函数调用直接通信。

**当前技术栈**:
- Python 3.13+, 进程内事件驱动 (asyncio.Queue)
- SQLite 持久化（每回测独立数据库文件）
- QMT/xtquant 作为行情与交易网关（Windows 本地 SDK）
- Tushare 作为辅助数据源

### 1.2 评估目标

本文档评估将单体拆分为微服务的可行性、优先级和潜在收益，提供可执行的拆分建议。核心问题是：**当前架构是否真的需要微服务化？如果需要，如何渐进式推进？**

## 2. 性能瓶颈分析

### 2.1 当前架构的性能特征

| 维度 | 现状 | 潜在瓶颈 |
|------|------|----------|
| **回测吞吐** | 单进程同步循环，逐日推进 | 大规模回测（全市场 x 多年）耗时显著 |
| **行情获取** | QMT 本地 SDK 调用，带透明缓存 | 缓存命中时无瓶颈；首次拉取受 QMT 限制 |
| **策略计算** | 进程内函数调用，零序列化开销 | 截面策略扫描全市场时 CPU 密集 |
| **ML 推理** | LightGBM 本地推理 | 批量推理时 CPU 密集，但无 I/O 瓶颈 |
| **交易执行** | QMT 本地 SDK，延迟极低（<1ms） | 受券商柜台延迟限制，非系统瓶颈 |
| **持久化** | SQLite WAL 模式 | 单写入者，无并发写入冲突 |
| **事件总线** | asyncio.Queue，进程内异步 | 无跨进程开销，但无法水平扩展 |

### 2.2 真实瓶颈识别

经过代码分析，当前系统的性能瓶颈集中在以下方面：

**瓶颈 1：回测计算密集（CPU-bound）**
- `BacktestAppService._run_unified_loop()` 逐日循环，每日执行策略信号生成 + 订单撮合 + 日终结算
- 截面策略（`CrossSectionalStrategy`）需扫描全市场快照，计算因子值、排名、过滤
- ML 因子挖掘（`FactorMiner`）和训练管道（`TrainingPipeline`）是典型的 CPU 密集型任务

**瓶颈 2：数据拉取 I/O（首次加载）**
- `IHistoryDataFetcher` 首次从 QMT/Tushare 拉取数据时存在网络 I/O
- 缓存机制已大幅缓解此问题（`data/{symbol}_{timeframe}.csv`）

**瓶颈 3：多策略并行回测**
- 当前 `run_backtest()` 通过 `strategies` 参数支持多策略，但仍是串行执行
- EventBus 模式为异步但仍在单进程内

### 2.3 瓶颈与微服务的关系

**关键结论**: 当前瓶颈主要是 **CPU 计算密集型**，而非 I/O 或服务间通信。微服务化引入的网络通信、序列化/反序列化开销反而可能降低回测性能。

| 瓶颈 | 微服务能否解决 | 更优方案 |
|------|---------------|---------|
| 回测 CPU 密集 | 能（水平扩展计算节点） | 多进程并行回测（更轻量） |
| 数据 I/O | 不能（瓶颈在数据源） | 缓存优化已足够 |
| 交易延迟 | 不能（受券商限制） | 保持本地 SDK |
| 多策略串行 | 能（独立策略服务） | 进程级并行即可 |

## 3. 拆分优先级评估

### 3.1 服务边界识别

基于 DDD 子域划分和当前代码结构，识别出以下候选服务：

#### 3.1.1 行情服务 (Market Service)

| 维度 | 评估 |
|------|------|
| **状态性** | 无状态（数据来自外部 QMT/Tushare，本地仅缓存） |
| **计算密集度** | 低（数据获取与格式转换） |
| **当前接口** | `IMarketGateway`, `IHistoryDataFetcher`, `IFundamentalFetcher` |
| **依赖的外部系统** | QMT (xtdata), Tushare API |
| **被依赖方** | strategy, backtest, risk, portfolio (通过接口) |
| **拆分难度** | 中等 — 接口已隔离，但 QMT SDK 是 Windows 本地库 |

**拆分收益**:
- 可独立部署为数据微服务，统一管理数据缓存与更新
- 支持多数据源聚合（QMT + Tushare + 其他）
- 为回测和实盘提供统一数据视图

**拆分风险**:
- QMT SDK 仅支持 Windows，限制部署环境
- 引入网络延迟，影响回测循环的紧密耦合调用

**评估结论**: **中优先级** — 接口隔离良好，但 QMT 本地依赖是硬约束。适合作为第二阶段拆分目标。

#### 3.1.2 交易服务 (Trade Service)

| 维度 | 评估 |
|------|------|
| **状态性** | 有状态（订单状态机、持仓、资金冻结） |
| **计算密集度** | 低（主要是状态管理与网关调用） |
| **当前接口** | `ITradeGateway`, `IAccountGateway` |
| **依赖的外部系统** | QMT (XtQuantTrader) |
| **被依赖方** | backtest, trading_app, order_executor |
| **拆分难度** | 高 — 订单状态机、T+1 结算、资金冻结逻辑紧密耦合 |

**拆分收益**:
- 交易服务独立部署，故障隔离
- 可支持多券商接入
- 订单状态管理独立演进

**拆分风险**:
- 订单状态机（`Order` 实体）与账户资产（`Asset`）强耦合，拆分需引入分布式事务
- T+1 结算依赖持仓状态，跨服务结算复杂度高
- 实盘交易对延迟极其敏感（毫秒级），网络通信增加延迟

**评估结论**: **低优先级** — 风险最高、收益有限。QMT SDK 本地调用延迟极低，拆分后网络延迟反而增加。仅在需要多券商支持时考虑。

#### 3.1.3 策略服务 (Strategy Service)

| 维度 | 评估 |
|------|------|
| **状态性** | 无状态（信号生成仅依赖输入数据） |
| **计算密集度** | 高（因子计算、截面排名、ML 推理） |
| **当前接口** | `BaseStrategy`, `CrossSectionalStrategy`, `IInferenceEngine` |
| **依赖的外部系统** | 行情数据（输入）、ML 模型（本地文件） |
| **被依赖方** | backtest, trading_orchestrator, signal_pipeline |
| **拆分难度** | 中等 — 策略信号生成已通过接口解耦 |

**拆分收益**:
- 策略计算可水平扩展（多策略并行）
- ML 推理可独立部署到 GPU 节点
- 策略热更新不影响交易服务

**拆分风险**:
- 策略信号生成依赖实时行情，需低延迟数据通道
- 因子计算需要全市场快照，数据传输量大
- 回测场景中策略与撮合紧密耦合（逐日循环）

**评估结论**: **高优先级（回测场景）** — 策略计算是主要 CPU 瓶颈，无状态特性天然适合拆分。但实盘场景中信号延迟要求高，需谨慎设计通信方式。

#### 3.1.4 风控服务 (Risk Service)

| 维度 | 评估 |
|------|------|
| **状态性** | 有状态（熔断器状态机、告警历史、暂停状态） |
| **计算密集度** | 中等（逐单风控 + 组合级风险计算） |
| **当前接口** | `RiskChain`, `CircuitBreaker`, `AlertEngine`, `BaseAnomalyDetector` |
| **依赖的外部系统** | 通知系统（通过 `IRiskNotifier`） |
| **被依赖方** | trading_orchestrator, backtest |
| **拆分难度** | 中等 — 风控通过接口解耦，但状态管理复杂 |

**拆分收益**:
- 风控独立部署，故障不影响交易执行（降级策略）
- 可独立演进风控规则和算法
- 支持多策略/多账户的统一风控视图

**拆分风险**:
- 逐单风控在交易链路上，增加延迟
- 熔断器状态需跨服务同步
- 异常检测需要访问全局状态

**评估结论**: **中优先级** — 风控的接口隔离良好（`RiskChain` 责任链模式），但逐单风控的延迟要求使其不适合远程调用。可考虑将风控规则引擎独立，但执行仍需本地代理。

### 3.2 优先级排序总结

| 优先级 | 服务 | 理由 |
|--------|------|------|
| **P0 (最高)** | 策略服务（回测） | 主要 CPU 瓶颈，无状态，天然可扩展 |
| **P1** | 行情服务 | 接口已隔离，统一数据管理价值高 |
| **P2** | 风控服务（规则引擎） | 接口解耦良好，但执行延迟要求高 |
| **P3 (最低)** | 交易服务 | 风险最高、延迟敏感、本地 SDK 依赖 |

## 4. 服务间通信方案

### 4.1 通信模式选型

| 通信模式 | 适用场景 | 延迟 | 可靠性 | 复杂度 |
|----------|----------|------|--------|--------|
| **gRPC** | 同步请求/响应，低延迟 | 低（<1ms 局域网） | 高 | 中 |
| **消息队列 (RabbitMQ/Kafka)** | 异步事件驱动，高吞吐 | 中（10-100ms） | 高 | 高 |
| **REST API** | 简单查询，跨语言兼容 | 中（10-50ms） | 中 | 低 |
| **共享数据库** | 强一致性，简单场景 | 低 | 高 | 低 |

### 4.2 推荐方案：gRPC + 消息队列混合

**实时路径（gRPC）**:
- 策略信号生成请求 → 策略服务
- 行情数据查询 → 行情服务
- 风控检查请求 → 风控代理

**异步路径（消息队列）**:
- 行情推送（行情服务 → 策略服务/风控服务）
- 风控事件通知（风控服务 → 通知服务）
- 交易执行结果回报（交易服务 → 策略服务/监控服务）
- 回测任务提交与结果返回

### 4.3 通信方案详细设计

```
                    ┌─────────────┐
                    │   接口层     │
                    │ (CLI/API)   │
                    └──────┬──────┘
                           │ gRPC
              ┌────────────┼────────────┐
              │            │            │
     ┌────────▼────┐ ┌─────▼─────┐ ┌───▼────────┐
     │  行情服务    │ │  策略服务  │ │  交易服务   │
     │ (无状态)     │ │ (无状态)   │ │ (有状态)   │
     └──────┬──────┘ └─────┬─────┘ └───┬────────┘
            │              │            │
            │    ┌─────────▼──────────┐ │
            └───>│    消息队列         │<┘
                 │  (事件总线)         │
                 └─────────┬──────────┘
                           │
                  ┌────────▼────────┐
                  │   风控服务       │
                  │  (规则引擎)      │
                  └─────────────────┘
```

### 4.4 数据序列化

- **gRPC**: 使用 Protocol Buffers 定义服务接口和消息格式
- **消息队列**: JSON 格式（易调试）或 Avro（高吞吐场景）
- **关键 Proto 定义**:
  - `MarketService`: `GetRecentBars`, `GetStockSnapshots`
  - `StrategyService`: `GenerateSignals`, `ComputeFactors`
  - `TradeService`: `PlaceOrder`, `GetPositions`, `GetAsset`
  - `RiskService`: `CheckOrder`, `EvaluateCircuitBreaker`

## 5. 渐进式拆分策略

### 5.1 核心原则

1. **优先拆分无状态服务** — 策略服务、行情服务天然无状态，拆分风险最低
2. **保持接口兼容** — 利用现有的 Protocol 接口（`IMarketGateway`, `ITradeGateway` 等），新增远程实现即可
3. **双模运行** — 新旧实现并存，通过配置切换，确保可回滚
4. **渐进式迁移** — 从回测场景开始（对延迟不敏感），逐步扩展到实盘

### 5.2 阶段规划

#### 第一阶段：回测场景策略服务化

**目标**: 将策略计算从回测主循环中解耦，支持远程策略执行

**具体步骤**:
1. 定义策略服务 gRPC 接口（Proto 文件）
2. 实现 `RemoteStrategyService` 包装器（包装现有策略实现）
3. 实现 `GrpcStrategyGateway`（application 层调用远程策略）
4. 回测主循环通过配置选择本地/远程策略执行
5. 验证回测结果一致性

**预期收益**:
- 策略计算可独立扩展到多节点
- ML 推理可部署到 GPU 节点
- 回测主循环不受策略计算阻塞

**风险控制**:
- 本地模式作为默认，远程模式需显式启用
- 回测结果必须与本地模式完全一致

#### 第二阶段：行情数据服务化

**目标**: 统一数据管理层，支持多数据源聚合

**具体步骤**:
1. 将 `MockMarketGateway`、`QmtMarketGateway`、`TushareHistoryDataFetcher` 等封装为行情服务
2. 实现行情服务的 gRPC 接口
3. 引入数据缓存层（Redis 或独立缓存服务）
4. 回测和实盘通过 gRPC 获取行情数据

**预期收益**:
- 数据缓存集中管理，避免重复拉取
- 多数据源透明切换
- 数据质量监控独立部署

#### 第三阶段：风控服务化

**目标**: 风控规则引擎独立部署，支持动态规则更新

**具体步骤**:
1. 将风控策略（`RiskChain` 中的各项策略）抽象为规则配置
2. 实现风控规则引擎服务（gRPC 接口）
3. 本地风控代理（`LocalRiskProxy`）缓存规则，本地执行逐单风控
4. 组合级风控（`PortfolioRiskService`）独立为远程服务

**预期收益**:
- 风控规则热更新
- 多策略/多账户统一风控视图
- 风控分析独立扩展

#### 第四阶段：交易服务化（可选）

**目标**: 多券商接入，交易服务独立部署

**前提条件**:
- 需要支持多券商
- 需要跨机器部署交易终端

**具体步骤**:
1. 定义统一交易接口（扩展 `ITradeGateway`）
2. 实现交易服务的 gRPC 接口
3. 各券商适配器作为交易服务的后端
4. 引入分布式事务或 Saga 模式处理订单状态

**风险警告**: 此阶段复杂度最高，仅在业务需求明确时推进。

## 6. 统一监控与日志方案

### 6.1 监控架构

```
┌─────────────┐  ┌─────────────┐  ┌─────────────┐
│  行情服务    │  │  策略服务    │  │  交易服务    │
│  指标采集    │  │  指标采集    │  │  指标采集    │
└──────┬──────┘  └──────┬──────┘  └──────┬──────┘
       │                │                │
       └────────┬───────┴────────┬───────┘
                │                │
         ┌──────▼──────┐  ┌─────▼──────┐
         │  Prometheus  │  │   Jaeger   │
         │  指标存储     │  │  链路追踪   │
         └──────┬──────┘  └─────┬──────┘
                │                │
         ┌──────▼──────────────▼──────┐
         │        Grafana              │
         │      可视化仪表盘            │
         └────────────────────────────┘
```

### 6.2 日志方案

**统一日志格式**:
```json
{
  "timestamp": "2026-06-01T09:35:00.123Z",
  "level": "INFO",
  "service": "strategy-service",
  "trace_id": "abc123",
  "span_id": "def456",
  "message": "Signal generated",
  "context": {
    "strategy": "dual_ma",
    "symbol": "000001.SZ",
    "direction": "BUY"
  }
}
```

**日志收集**:
- 各服务输出结构化 JSON 日志到 stdout
- Filebeat/Fluentd 收集并转发到 Elasticsearch
- Kibana 提供日志查询和分析

### 6.3 关键监控指标

| 服务 | 关键指标 | 告警阈值 |
|------|----------|----------|
| 行情服务 | 数据延迟、缓存命中率、QMT 连接状态 | 延迟 > 5s、缓存命中 < 80% |
| 策略服务 | 信号生成延迟、因子计算耗时、ML 推理耗时 | 信号延迟 > 10s |
| 交易服务 | 订单提交延迟、成交率、拒绝率 | 拒绝率 > 5%、延迟 > 1s |
| 风控服务 | 风控检查延迟、熔断触发次数、告警数量 | 熔断触发 |
| 通知服务 | 通知发送成功率、发送延迟 | 成功率 < 95% |

### 6.4 分布式追踪

- 使用 OpenTelemetry SDK 实现跨服务链路追踪
- 关键链路: 信号生成 → 风控检查 → 订单提交 → 成交回报
- 追踪 ID 贯穿整个交易循环，便于问题定位

## 7. 技术选型建议

### 7.1 服务框架

| 组件 | 推荐方案 | 备选方案 |
|------|----------|----------|
| **RPC 框架** | gRPC (grpcio) | Thrift |
| **消息队列** | RabbitMQ | Kafka (高吞吐场景) |
| **服务发现** | Consul | etcd |
| **配置中心** | Consul KV + 环境变量 | Nacos |
| **API 网关** | Kong | Traefik |
| **容器编排** | Docker Compose (初期) → Kubernetes (规模化) | - |
| **日志收集** | ELK Stack (Elasticsearch + Logstash + Kibana) | Loki + Grafana |
| **指标监控** | Prometheus + Grafana | Datadog |
| **链路追踪** | Jaeger (OpenTelemetry) | Zipkin |

### 7.2 Python 微服务框架

- **gRPC**: `grpcio` + `grpcio-tools` (Proto 代码生成)
- **异步支持**: `asyncio` + `grpcio` (async server)
- **消息队列客户端**: `pika` (RabbitMQ) 或 `confluent-kafka`
- **健康检查**: 标准 `/health` 端点

## 8. 风险与约束

### 8.1 硬约束

1. **QMT SDK Windows 依赖**: xtquant 仅支持 Windows，行情和交易网关必须部署在 Windows 环境
2. **本地交易终端**: QMT 交易模块需要本地 MiniQMT 终端运行
3. **数据源限制**: Tushare 有 API 调用频率限制

### 8.2 拆分风险

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| 网络延迟增加 | 实盘交易延迟敏感 | 逐单风控保持本地代理 |
| 分布式事务复杂 | 订单状态一致性 | 初期避免拆分交易服务 |
| 运维复杂度上升 | 需要容器编排能力 | 从 Docker Compose 开始 |
| 调试困难 | 跨服务问题定位 | 分布式追踪 + 统一日志 |
| 数据一致性 | 回测结果可重复性 | 严格版本控制 + 确定性执行 |

### 8.3 不建议拆分的场景

1. **回测核心循环** — `BacktestAppService` 的逐日推进逻辑（行情 → 策略 → 撮合 → 结算）是紧密耦合的事务性操作，拆分后序列化开销可能抵消并行收益
2. **实盘交易链路** — 信号 → 风控 → 下单的链路对延迟极敏感，保持进程内调用更优
3. **SQLite 持久化** — 当前每回测独立数据库的模式简单可靠，无需分布式数据库

---

## 9. 结论与建议

### 9.1 核心结论

1. **当前单体架构基本合理** — DDD 分层 + 接口隔离为未来拆分预留了良好基础
2. **微服务化非紧迫需求** — 当前系统规模和性能需求尚未达到必须拆分的程度
3. **渐进式拆分可行** — 优先拆分无状态的策略计算服务（回测场景），风险最低、收益最明确
4. **保持实盘链路简洁** — 实盘交易的核心链路（信号 → 风控 → 下单）应保持进程内调用

### 9.2 行动建议

| 时间线 | 行动 | 优先级 |
|--------|------|--------|
| **短期 (1-3 月)** | 优化现有单体架构：进程级并行回测、缓存优化 | 高 |
| **中期 (3-6 月)** | 策略服务 gRPC 化（回测场景验证） | 中 |
| **长期 (6-12 月)** | 行情服务化、风控规则引擎独立 | 低 |
| **按需** | 交易服务化（仅在多券商需求明确时） | 最低 |

### 9.3 替代方案

如果主要诉求是回测性能提升，以下方案比微服务拆分更轻量：

1. **多进程并行回测**: 利用 `multiprocessing` 并行运行多个策略/标的的回测
2. **异步优化**: 将 EventBus 模式作为默认，充分利用 asyncio 的 I/O 并发
3. **数据预加载**: 将全量数据预加载到内存，减少 I/O 等待
4. **NumPy/Pandas 向量化**: 在基础设施层利用 NumPy 加速因子计算（domain 层保持纯净）
