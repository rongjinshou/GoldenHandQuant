## 一、背景与主线定位

漏斗主线（给 ¥146k 找可验证 edge）的纯因子探索已挖尽：单因子 + 多因子组合均**无可投 long_only alpha**，找到的全是防御 beta（size/value/quality/低波），集体在 OOS 牛市跑输。唯一例外是 B2 的转机 —— **F01 size + 中证 1000 趋势闸择时**：全窗 Sharpe 0.76→1.51、**OOS Sharpe 0.97→1.73**、两窗回撤均约减半，闸效在 OOS 泛化（非样本内躲崩）。这是漏斗**首个 OOS 验证的可投候选**，本质是「防御型 size 暴露 + 趋势择时」，**不是**跑赢市场的新 alpha。

**主线决策（2026-06-26）**：接受 F01+趋势闸 为当前唯一 OOS 验证的可投候选，**推向小资金 live**。诚实定位 —— 目标是「用小资金验证这套候选在真实摩擦下能否复现回测画像（低回撤的小盘 beta 替代）」，不是赚 alpha。

## 二、范围与决策留痕

本 Spec 经 4 轮决策对齐，范围锁定如下：

| 维度 | 决定 | 理由 |
|---|---|---|
| 范围边界 | 接入实盘链路 + dry-run 影子盘验「信号一致性」；**真单 live 切下一 Spec** | 真单前先确认画像在真实数据流复现；高风险真单单独 Spec 单独审 |
| 板块 | **只主板**（60/000/001） | 用户真实约束（账户权限）；且天然兼容现有实盘主板白名单 |
| 资金/调仓 | ¥146k 全额 + 严格复刻周二调仓 + 1/4 月空仓 | 对齐真账户与回测口径，跟踪误差才可比 |
| 验证重心 | 只验**信号一致性**（实盘信号 == 回测信号，逐日一致，无前视/无漂移） | 接入正确性是地基；执行可行性 / 长期画像跟踪不在本 Spec |
| 入口 | **先验主板域 F01 画像是否成立（gate）** | 只主板改变选股域，F01 原画像依赖创业板微盘，主板域成不成立**未验证** |

**核心张力（已识别）**：「只主板」×「信号一致性」×「F01 原画像依赖创业板微盘」三者冲突 —— 实盘只主板选股域 ≠ 回测全市场域，目标组合**天然不一致**（非接入 bug）。处置：把信号一致性的对照基准改成「**只主板回测**」，并先用 gate 验证主板域 F01 是否成立，再决定建不建影子盘。

## 三、两阶段结构

```
阶段 0  主板域 F01 验证关口（小研究，本 Spec 直接执行，不走 plan）
        └─ pass → 阶段 1 ;  fail → 停下重议板块（影子盘不建）
阶段 1  dry-run 影子盘（条件性，gate pass 才建，gate 后单独 brainstorm→plan→实现）
```

- **阶段 0 — 主板域 F01 验证关口**：回答「F01+趋势闸 在只主板域，IS/OOS 画像是否仍可投（保留防御特征）」。小研究、复用现成脚本、纯回测、不碰实盘 —— 像 B1/B2 那样直接做。
- **阶段 1 — dry-run 影子盘**：把 F01+趋势闸 截面引擎接入实盘 `auto-trade` 信号路径，dry-run 验证实盘目标组合 == 只主板回测同日输出。强依赖阶段 0 结果，故 gate 后再细化。

## 四、阶段 0 设计决策（DD）

### DD-1 · 复用 `b2_trend_gate` 套路，universe 加主板过滤
新脚本 `scripts/mainboard_f01_gate.py`，骨架同 `b2_trend_gate_ab.py` / `b2_trend_gate_oos.py`：`build_backtest_cross_section("DuckDBHistoryDataFetcher", ...)`（DuckDB 离线全市场）→ **universe 主板过滤** → 装 bars 入 `MockMarketGateway` → 趋势闸 ON 跑 IS/OOS。复用现成 `BacktestAppService` / `MockTradeGateway` / `EqualWeightSizer` / `PerformanceEvaluator`。几乎零新工程。

### DD-2 · 主板定义复用实盘白名单口径（关键，对齐信号一致性）
阶段 0 的「主板」过滤口径**必须 == 实盘 live order guard 的主板白名单判定**（runbook 载明拒单 `reason="不在 v1 允许范围"`，60/000/001）。**不在脚本手写前缀后各自漂移**：
- **优先**：执行时 `grep "允许范围"` 定位实盘那道白名单，若其判定已是（或可低成本抽成）一个可导入的纯函数 `is_mainboard(symbol)`，gate 脚本与实盘**共用一处定义**（呼应 [[reuse-not-recompute]]）。
- **退路**：若实盘判定深埋 service 不易复用，gate 脚本按**完全相同口径**（60/000/001 前缀）实现，并登记债「阶段 1 抽取共享 `is_mainboard`」—— 不让重构阻塞 gate。

### DD-3 · 对照口径：只主板回测 + 全市场基线 + 只主板等权基准
- **主跑**：只主板 F01+趋势闸（top20，2021-01-01→2026-06-11，split 2024-06-30）。
- **基线对照**：全市场 F01+趋势闸（已知 OOS Sharpe 1.73）—— 量化主板约束让画像衰减多少。
- **基准**：只主板等权覆盖池（同 F01 可投性报告 §一口径，costless）—— 判防御是否保留。

### DD-4 · gate 判据 = 相对防御画像保留（非绝对阈值）
主板域 **OOS（2024-06→2026）** 满足以下「防御画像」即 **pass → 进阶段 1**：
1. **最大回撤显著低于只主板等权基准**（核心 —— F01 的卖点是回撤腰斩）；
2. **趋势闸 ON 较 OFF 的 Sharpe 抬升 / 回撤下降在主板域保留**（趋势择时增益不依赖创业板）。

反之，若主板域**趋势闸增益蒸发**或**回撤不再优于基准** → **fail → 停下重议板块**（放宽到创业板 / 接受只主板不是 F01 部署域）。不用武断绝对阈值，因 F01 定位是防御 beta（相对画像），不是绝对收益 alpha。

### DD-5 · 趋势闸 ON（部署对象是 F01+趋势闸，非裸 F01）
gate 主跑趋势闸 **ON**（中证 1000 `close<MA20` 阻断买入），因为要影子盘/部署的就是 F01+趋势闸 组合。同时附 **OFF 对照**量化主板域趋势闸净增益（同 B2 口径，ON−OFF 差值天然对冲生存者偏差）。

## 五、阶段 1 骨架（gate pass 后细化，此处仅定方向）

- **接入点**：实盘 `auto-trade` 信号生成路径当前调用 `dual_ma`（时序、固定 symbols）；阶段 1 让它能调度 `MicroValueStrategy` 截面策略 + 趋势闸（`CrossSectionalStrategyRunner` 已在回测验证，需接到 live 信号路径）。
- **四冲突处置**：① 截面策略接入 —— 核心工程，gate 后细化；② 主板白名单 —— 已限只主板，**天然兼容（消解）**；③ 单笔预算 —— ¥7k/只，废掉 ¥1500 闸，重标定；④ 调仓日历 —— 周二 + 1/4 月空仓。
- **验证**：dry-run 目标组合逐日 == 只主板回测同日输出（无前视 / 无漂移）。
- **非目标**：不碰真单（下一 Spec）；不动 factor-test / 重判；不改驾驶舱 UI；不删 `dual_ma` 路径。

## 六、风险与偏置（诚实校准）

- **生存者偏差（B1 仍在）**：退市股缺失，主板域**绝对收益**仍偏乐观上界；但 gate 判据是**相对防御画像**（ON−OFF / vs 同宇宙基准），生存者偏差对冲，裁定稳健。
- **单 OOS regime**：OOS 仍是单一小盘牛，「防御」结论单样本。
- **主板域样本变薄**：只主板最小市值股数量 < 全市场，top20 截面可能更集中 / 噪声更大 —— gate 报告须标注主板域宇宙规模与最小一档画像。
- **主板「最小市值」≠ 创业板微盘**：主板小盘市值普遍更大，F01 的 size 暴露在主板域更弱，**画像衰减是预期内** —— gate 正是来量化它，而非假设它不变。

## 七、验收标准（阶段 0）

1. `scripts/mainboard_f01_gate.py` 跑通，输出主板域 F01+趋势闸 IS/OOS（ON/OFF）+ 只主板等权基准对照，全程 `$WIN_PYTHON` 离线、只读 `market.duckdb`。
2. 主板过滤口径与实盘白名单一致（共用 `is_mainboard` 判定，或同口径 + 登记抽取债）。
3. 产出 gate 裁定：**pass**（防御画像保留 → 进阶段 1）/ **fail**（→ 重议板块），结论写入同目录 `*-report.md` + 更新 [[factor-funnel-status]] 记忆。
4. 阶段 1 仅在 gate pass 后另起 brainstorm→plan，不在本 Spec 内实现。
