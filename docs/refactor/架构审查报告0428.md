## 二、遗漏的重要缺陷

以下缺陷在当前版本的 19 项列表中未被纳入，但在代码审查中被识别：

### 遗漏-1: 除权除息/分红事件未建模 (高)

**说明**: A股每年发生大量除权除息事件（现金分红、送股、转增、配股）。当前回测引擎完全不处理：

| 事件类型 | 影响 | 当前状态 |
|---------|------|---------|
| 现金分红 | 应增加 `available_cash` | 未建模 |
| 送股/转增 | 持仓数量应增加对应比例 | 未建模 |
| 配股 | 需缴款，持仓数量增加 | 未建模 |
| 除权除息日跳空 | 复权价连续，但不复权价跳空 | 仅依赖单一价格字段 |

报告仅在讨论 #5（复权对齐）时间接涉及，但 **未将分红事件处理列为一个独立缺陷**。这是 A 股回测中与 #1（未来函数）同等重要的建模缺失。前复权保持价格曲线连续性，但 **真实现金流**（分红入账）被完全忽略，导致长期回测中权益曲线偏离真实。

**建议优先级**: P0（业务类），与 #5 并列。

### 遗漏-2: 回测缺少基准对比 (中)

**说明**: `BacktestReport` 不包含任何基准对比数据（如沪深300、中证500同期表现）。无法评估策略是否产生 Alpha。`PerformanceEvaluator` 不计算 Alpha、Beta、信息比率等相对指标。

**建议优先级**: P2。

### 遗漏-3: 策略参数无版本化/实验追踪 (中)

**说明**: DualMaStrategy 的 MA5/MA10 参数硬编码，无参数版本管理。多次回测对比不同参数时，无法追溯哪组参数产生了哪个报告。这对策略迭代至关重要。

**建议优先级**: P2。

### 遗漏-4: 回测无断点续跑/状态序列化 (低-中)

**说明**: 回测状态（持仓、资金、订单历史）无法持久化。长回测中断后必须从头开始。`BacktestAppService` 无 `save_checkpoint()` / `resume_from_checkpoint()` 方法。

**建议优先级**: P3。

### 遗漏-5: 日内回测不支持交易时间约束 (低)

**说明**: 虽然当前以日线回测为主，但 A 股有 9:30-11:30 / 13:00-15:00 的交易时段约束和 9:15-9:25 集合竞价。若未来扩展分钟级回测，这些时间约束缺少建模框架。

**建议优先级**: P3（远期），报告方向中提及分钟级但未展开。

### 遗漏-6: Mock撮合无盘口深度模拟 (低)

**说明**: `mock_trade.py` 的流动性限制仅为 `成交量 * 10%`，不模拟订单簿深度。大单应以滑点递增方式成交（连续吃掉多个价位），而非固定滑点率。

**建议优先级**: P3，属于回测精度的深度优化。

## 三、架构图层次关系验证

### 3.1 层次结构

报告中的架构图正确描述了 DDD 四层结构：

```
interfaces (用户接口层)
    ↓
application (应用编排层)  
    ↓
domain (纯领域层)
    ↓
infrastructure (基础设施层)
```

依赖方向 `interfaces → infrastructure → application → domain`（由外向内）与 CLAUDE.md 规定一致。

### 3.2 实体与接口映射验证

| 报告声称 | 代码实际存在 | 一致 |
|---------|------------|------|
| Order 状态机 (6种状态) | `order.py` — 确认为 CREATED/SUBMITTED/PARTIAL_FILLED/FILLED/CANCELED/REJECTED + PARTIAL_CANCELED | 报告少列 PARTIAL_CANCELED |
| IMarketGateway = Protocol | `market_gateway.py:6` ✅ | 是 |
| Position.available_volume (T+1) | `position.py:25` ✅ | 是 |
| Asset 区分 available_cash/frozen_cash | `asset.py:24-25` ✅ | 是 |
| 双均线策略 (MA5/MA10) | `dual_ma_strategy.py` ✅ | 是 |
| FixedRatioSizer | `fixed_ratio_sizer.py` ✅ | 是 |

### 3.3 架构图小问题

1. **Order 状态机**：报告列出 6 种状态，实际代码中 `OrderStatus` 还包含 `PARTIAL_CANCELED`（订单取消时若已部分成交则转为此状态）。报告状态机描述省略了此状态。

2. **费率常量位置**：报告 1.4 节说费率"已内置于 MockTradeGateway 和 DailySettlementService"，但 `DailySettlementService` 仅计算撤单解冻时的费用估算，实际费率定义在 `MockTradeGateway` 中。两处各有独立常量声明，有重复。

3. **IMarketGateway 接口**：报告列出 `get_recent_bars()`, `get_all_timestamps()` 两个方法。`get_recent_bars()` 的 `timeframe` 参数类型在代码中为 `Timeframe`（枚举），但调用时 mock_trade.py:91 传入了字符串 `"1d"`，这在 Protocol 类型检查下可能产生警告。
