**数据流 (回测主循环)**:
`MarketGateway.set_current_time(t)` 
→ `MarketGateway.get_recent_bars()` → `Strategy.generate_signals()`
→ `PositionSizer.calculate_target()` → `Order` 构造
→ `TradeGateway.place_order()` [含滑点/流动性/费用模拟]
→ `DailySettlementService.process_daily_settlement()`
→ `_record_snapshot()` → `PerformanceEvaluator.evaluate()` → `BacktestReport`

### 1.1 关键领域实体

| 实体               | 所在子域 | 职责                                                         |
| ------------------ | -------- | ------------------------------------------------------------ |
| **Order**          | trade    | 订单状态机：CREATED → SUBMITTED → (PARTIAL_FILLED) → FILLED / CANCELED / REJECTED / PARTIAL_CANCELED。买入量必须为 100 的整数倍 |
| **Position**       | account  | 区分 total_volume 和 available_volume（T+1：当日买不可卖）   |
| **Asset**          | account  | 区分 available_cash 和 frozen_cash，下单冻结，成交/撤单解冻  |
| **Bar**            | market   | OHLCV K 线数据，带 Timeframe 枚举                            |
| **Signal**         | strategy | 策略生成的买卖信号，含置信度评分                             |
| **BacktestReport** | backtest | 回测聚合报告：收益率、最大回撤、胜率、权益曲线等             |

### 1.2 接口隔离模式

| 接口                     | 类型     | 定义方法                                                     |

| `IHistoryDataFetcher`    | ABC      | `fetch_history_bars()`                                       |

### 1.4 A 股交易成本（已内置）

- 佣金：双向万 2.5，最低 5 元
- 印花税：仅卖出，千 0.5
- 过户费：双向，十万分之一
- 滑点：买入上浮 0.1%，卖出下浮 0.1%
- 流动性限制：单笔不超过当日成交量 10%

## 二、当前架构与业务逻辑缺陷深度剖析

### 2.1 致命交易逻辑缺陷（高危，阻碍系统可用）

| #    | 缺陷                                 | 严重度 | 深度解析                                                     |
| ---- | ------------------------------------ | ------ | ------------------------------------------------------------ |
| 1    | **未来函数与偷价（Lookahead Bias）** | 极高   | 策略计算均线切片（如 `bars[-5:]`）包含了 T 日数据，且回测循环立马使用 **T日收盘价** 进行 `place_order` 撮合。在真实A股中，确认 T 日收盘时无法以该价买入。将导致虚假完美收益曲线。 |
| 2    | **仓位管理（Sizer）被完全架空**      | 极高   | 尽管架构图包含 `PositionSizer`，但 `DualMaStrategy` 直接硬编码了 `target_volume=100`。策略层越权插手资金管理，导致系统彻底丧失资金风险暴露控制和组合再平衡能力。 |
| 3    | **负资金透支漏洞（Mock击穿）**       | 高     | Mock 引擎 `_simulate_fill` 中，若滑点导致实际成本超预期，直接暴力执行 `available_cash -= diff`。A 股现金不允许为负，该透支行为将引起后续净值与收益率雪崩式错误。 |
| 4    | **限价单（LIMIT）语意失效**          | 高     | `Order` 默认设为 `OrderType.LIMIT`，但模拟撮合直接用 `exec_price = ref_price * (1 + 滑点)` 强行成交，完全绕过了与当日 High/Low 极值的比对，使订单状态机失去挂单/撤单仿真意义。 |

### 2.2 A 股业务建模层面缺陷

| #    | 缺陷                   | 严重度 | 说明与深度解析                                               |
| ---- | ---------------------- | ------ | ------------------------------------------------------------ |
| 5    | **账户体系不复权对齐** | 高     | 规范要求 K 线复权、账本真实。当前 `MockTradeGateway` 错误地直接用 K 线**前复权价**扣费（如某股十年前前复权5元，真实价格20元），这会凭空制造出极度夸张的虚假购买力。必须引入 `unadjusted_close` 专供账本结算。 |
| 6    | **无涨跌停限制模拟**   | 高     | 回测未模拟 A 股 ±10%（科创/创业 ±20%）板，涨停无法买入、跌停无法卖出均未建模。 |
| 7    | **无停牌/ST 处理**     | 高     | 数据加载和信号生成未过滤停牌或 ST 股票，导致在不可交易标的上触发虚假信号。 |
| 8    | **单账户模型硬编码**   | 中     | 所有实体写死 `account_id="MOCK_ACCOUNT"`。在多策略并行回测或母子账户管理时，买卖单会混入同一个字典，导致底层资金和持仓严重串户。 |

### 2.3 架构设计层面缺陷

| #    | 缺陷                              | 严重度 | 说明与深度解析                                               |
| ---- | --------------------------------- | ------ | ------------------------------------------------------------ |
| 9    | **无风控策略链（Risk Chain）**    | 中     | `OrderService` 目前只注入单一风控。缺乏责任链模式（Chain of Responsibility），无法实现：查停牌（黑名单）→ 查集中度（防单标超额）→ 查日内回撤（熔断）的串联拦截。 |
| 10   | **BacktestReport 贫血模型**       | 中     | 核心回测指标（最大回撤、夏普等）的过程化代码散落在 `PerformanceEvaluator` 中，`BacktestReport` 退化成了纯 DTO。应使用 `@dataclass(frozen=True)` 将计算行为内聚为 property。 |
| 11   | **无 DI 容器 / 模块空心化**       | 中     | 依赖在 `main.py` 纯手工组装。且规划的 `event_bus` / `persistence` / `ml_engine` 目前均只有空 `__init__.py`，架构未能兑现异步解耦与持久化能力。 |
| 12   | **回测与实盘 MarketGateway 分裂** | 中     | `IMarketGateway` 包含了 `get_all_timestamps()`（仅回测用），导致实盘类直接抛 `NotImplementedError`，接口被回测需求污染。 |
| 13   | **Portfolio 子域单薄**            | 中     | 目前仅有 `FixedRatioSizer` 骨架，缺少凯利公式、风险预算等算法，且未被真正调用。 |
| 14   | **Strategy 无多策略支持**         | 中     | AppService 只注入单一 Strategy，无法进行多策略组合或权重对比。 |

### 2.4 代码质量层面缺陷

| #    | 缺陷                 | 严重度 | 说明                                                         |
| ---- | -------------------- | ------ | ------------------------------------------------------------ |
| 15   | **测试覆盖率不足**   | 高     | `PerformanceEvaluator` 仅1个测试；无集成测试；无 `conftest.py` / `pytest.ini`。 |
| 16   | **硬编码严重**       | 中     | QMT 路径、session_id 均硬编码，无 YAML/JSON 配置文件系统接入。 |
| 17   | **错误处理粗糙**     | 中     | 存在宽泛的 `except Exception`；Domain 层缺乏诸如 `InsufficientFundsError` 专用异常。 |
| 18   | **xtquant SDK 内嵌** | 中     | SDK 二进制文件直接堆在 `src/` 下，未剥离，且破坏了跨平台纯净性。 |
| 19   | **无日志/可观测性**  | 中     | 回测主循环如黑盒，无进度条，无结构化埋点记录，排查极其困难。 |

## 三、后续规划方向与 Roadmap

### 优先修复区（紧急）
- **斩断未来函数**：引入时间隔离，T 日生成信号并压入队列，T+1 日获取新行情后执行撮合。
- **重塑仓位职责**：将 `Signal` 的 `target_volume` 剥离，通过统一的 `PositionSizer` 计算股数。
- **修复账户击穿与限价漏斗**：添加可用资金的硬熔断防御；完善 Mock 引擎对 `OrderType.LIMIT` 的极值校验。
- **真实价域穿透**：引入 `unadjusted_close` 隔离复权价格与真实账本结算。

### 方向 A：夯实基础
- 补测试覆盖率（BacktestAppService、PerformanceEvaluator、集成测试）。
- 引入 YAML/JSON 配置文件系统（config/ 目录落地）及消除硬编码。
- 引入 `pyproject.toml` 及统一的依赖管理。
- 完善日志与可观测性（结构化日志、tqdm 回测进度条）。

### 方向 B：业务仿真深度
- 涨跌停板模拟（涨停无法买入、跌停需排队）。
- 停牌/ST 股票过滤（数据加载 + 信号生成双向拦截）。
- 多账户支持（引入 `AccountRepository` 实现分仓管理与物理隔离）。
- 落地风控责任链（单标的上限、行业集中度、日内回撤熔断）。
- 激活并丰富仓位管理算法（凯利公式、等权重再平衡）。

### 方向 C：架构能力扩展
- 重构领域模型为充血模型（如 `BacktestReport` 值对象化）。
- 实现进程内事件总线（EventBus，基于 `asyncio.Queue` 实现事件溯源）。
- 实现持久化层（SQLite 存储订单/持仓/快照流水）。
- ML 引擎集成（CatBoost / XGBoost 模型推理封装）。
- FastAPI 实盘/回测 API 接口，以及研报导出引擎（HTML/PDF）。

### 综合优先级建议

| 优先级 | 类别         | 具体事项                                              |
| ------ | ------------ | ----------------------------------------------------- |
| **P0** | **高危逻辑** | **修复未来函数、Sizer越权、负资金击穿、Mock限价失效** |
| P0     | 基础         | 引入 pyproject.toml + 依赖管理                        |
| P0     | 业务         | 真实价格账本隔离 (复权解耦) + 涨跌停/停牌/ST 过滤     |
| P1     | 基础         | 配置文件系统 + 消除硬编码 + 补测试覆盖率              |
| P2     | 业务         | 多账户解耦 + 风控责任链落地 + 充血模型改造            |
| P2     | 基础         | 日志/可观测性完善                                     |
| P3     | 能力         | 事件总线、持久化层、ML 引擎集成、多策略并行 / API     |

### 5.2 报告遗漏的重要缺陷

以下是通过代码审查发现的、报告未覆盖的关键缺陷：

#### 遗漏 A：总资产计算存在循环依赖（严重度：高）

`backtest_app.py:218` 在 `_record_snapshot` 中 `asset.total_asset = total_asset` 直接覆写了 Asset 实体的总资产。但 `FixedRatioSizer.calculate_target()` 使用 `asset.total_asset * self.ratio` 计算预算（`fixed_ratio_sizer.py:30`）。如果当日产生了交易盈亏导致 total_asset 变化，下一次策略循环的预算计算会基于被覆盖后的值。这导致仓位预算与资产快照之间存在不透明的耦合。

#### 遗漏 B：日内交易多次信号冲突未处理（严重度：中）

`backtest_app.py` 的回测循环在一次 tick 内可以生成多个同标的信号（买+卖同时出现），每个信号都独立调用 `sizer.calculate_target()` 和 `place_order()`。如果 DualMaStrategy 和其他策略同时运行，可能出现"先买后卖"同一标的的荒诞交易序列，且资金和持仓状态在循环中途已经变化，但循环内未重新获取。

#### 遗漏 C：除权除息事件未建模（严重度：中）

A 股分红送股时，股价会除权除息跳空下跌，持仓市值不变但账户收到现金分红/红股。当前系统完全不处理：
- 前复权 K 线掩盖了除权缺口，但账本层面没有对应现金入账
- 如果使用不复权价格结算（修复 #5 后），又会因除权日价格跳空产生虚假盈亏

#### 遗漏 D：科创板/创业板差异化规则未区分（严重度：中）

报告 #6 提到 ±10%/±20% 涨跌停差异，但代码库中 `bar.symbol` 仅存储纯代码（如 "600000.SH"），没有任何地方根据代码前缀判断板块归属。科创板的上市首日不限涨跌幅、盤后固定价格交易等特殊机制完全未涉及。

#### 遗漏 E：QMT 实盘网关缺少连接状态监控与重连机制（严重度：中）

`qmt_trade.py:50-61` 中 `connect_result` 非 0 仅打日志，不抛异常，后续所有操作在无效连接上静默失败。`qmt_market.py` 无任何连接管理代码。

#### 遗漏 F：交易成本费率硬编码且无法按市场/品种差异化（严重度：低）

费率常量在多个文件中重复定义（`mock_trade.py:25-35` 和 `settlement_service.py:17-19`），违反 DRY 原则。且不同品种费率不同（ETF 免印花税、可转债费率不同），当前没有按品种路由的机制。

#### 遗漏 G：Asset 实体 frozen_cash 与 total_asset 一致性未保证（严重度：中）

`asset.py` 未在审查中展开验证，但 `mock_trade.py:215` 手动修改 `self.asset.total_asset -= (commission + tax + transfer_fee)`，而不是通过 Asset 的方法内聚更新，绕过了实体的不变量保护。`backtest_app.py:218` 同样的问题。

### 5.7 优化 Plan 审查要点（针对交叉验证任务的输入）

以下发现供交叉验证专家参考：

1. **Plan 与报告缺陷的覆盖对应**：Plan Phase 0-3 覆盖了报告中 19 个缺陷中的 15 个。未直接覆盖的：#2 (Sizer 职责——Plan 未提改造策略让 Signal 携带仓位建议或增强 FixedRatioSizer 以考虑已有持仓增量)、#10 (充血模型——Plan 未提将指标计算移入 BacktestReport)、#17 (错误处理——Plan 未提创建专用异常类)

2. **Plan 代码示例中的潜在问题**：
   - Task 0.1 Step 0.1.1 的代码将 `execution_prices` 替换为 `bars[-1].open`，这改变了整个撮合价格体系，但性能评估中的 snapshot 仍用 close 估值——open 成交 + close 估值的组合是合理的，但 Plan 未解释这一设计决策
   - Task 0.3 Step 0.3.2 的 LIMIT 极值校验逻辑有瑕疵：买单限价低于最低价时应拒单，但实际场景中买单限价高于最高价才是真正的"当天不可能成交"。正确的校验应为：买单限价 < bar.low → 当天最低价都高于你的出价，不可能成交。但还有一种情况：涨停封板时即使限价=涨停价也无法买入（需排队）。Plan 的实现未考虑涨停封板场景。

3. **Phase 顺序建议**：Phase 0 的 4 个 Task 之间存在依赖——Task 0.4 (unadjusted_close) 应在 Task 0.3 (LIMIT 极值校验) 之前，因为极值校验依赖正确的结算价格基准。
