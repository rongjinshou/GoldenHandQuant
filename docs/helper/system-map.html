第七区 · DOMAIN MODEL

域模型 —— 九个子域，一个决策核

domain 层 11.2k 行 / 228 文件，只依赖标准库 + numpy/pandas/scipy（AST 门禁强制）。 最重要的结构事实：回测和实盘共用同一个决策核（StrategyRunner，0628 归一）——回测里跑的逻辑就是实盘里跑的逻辑。

trade 交易 核心

Order 状态机（7 态，BUY 须 100 整数倍）· 六道盘前闸纯函数（单一事实源，三条下单路共用）· 交易日历/时段 · 订单轮询 helper · 执行质量统计

account 账户 核心

Asset 资金守恒（available/frozen 冻结三方法）· Position T+1 双量（total/available 分离，日终 settle 释放）· 日终结算服务

strategy 策略 最大子域

4 策略 + 5 过滤器 + 30+ Factor 实现 · factor_test 检验机器（DSL 解析→IC/分层/衰减/中性化→7 道闸判决；对象式与向量化双路逐位等价）· 截面装配器 · 策略池（未接线）

risk 风控

订单级责任链（单票 30%/总仓 80%）· 信号级止损（硬止损 3%/涨停破板）· 趋势闸（指数<MA20 禁买）· 熔断器（跨进程续命）· 告警引擎 · 组合风控子树（~900 行零引用）

market 行情

Bar/Quote/StockSnapshot（三段式，KNOWN_FIELDS 校验源）· 特征引擎 v1（29 列向量化，T-1 约定）· 涨跌停计算（主板10%/双创20%/ST5%）· ST 前缀单一事实源 · as-of 状态注册表

backtest 回测

BarWindow 前视边界（信息≤T-1，执行=T 开盘）· 绩效评估（Sharpe/Sortino/Calmar）· 对比报告（相关矩阵+低相关组合推荐）· IBacktestBroker 组合接口防碎片化

portfolio 仓位

EqualWeightSizer（等权+1% 现金 buffer；空目标池=清仓语义；负成本防线）· FixedRatioSizer（bar 策略用）· OrderTarget

notification 通知

优先级队列（EMERGENCY 即发/INFO 批量）· 滑动窗口去重 · 静默时段 · 脱敏——经 NotificationHub 聚合接入 auto-trade

common 公共

领域事件基类 · AuditService（append-only 审计，动作全留痕）

决策核解剖 CrossSectionalStrategyRunner.evaluate — 每个交易日发生什么

回测里这条链由 BacktestAppService 驱动（Mock 撮合），影子/实盘由 AutoTradeAppService 驱动（后接三层防线 + 留痕）。同一份代码，两个宿主。

第八区 · WHERE IT ROTS

乱区诊断 —— 你说的"太乱"，具体乱在哪

这一区是本图的重点。"乱"不是笼统的感受，它有七个具体位置。按严重度排，每条都标了证据与建议动作。 关键判断：核心链路（决策核、防线、留痕、门禁）其实是干净的——乱在边缘：没提交的工作区、没接线的子系统、重复的装配、失真的文档。

① 工作区领先版本库整整一个迭代 git status = 146 改动 + 51 未跟踪 · 最近 4 个提交都叫「例外: 借用提交代码」

这是最大的乱源，也是最容易解决的。影子盘编排、ST 诚实债、市值口径 MC-1、交易日历、tushare 资产、因子挖掘 E9、六西格玛演进—— 这些已经写完、测过、在生产里跑了好几天的代码，全部只存在于工作区。6 个 docs/feat/ 专题目录、11 个新脚本、9 个新 src 模块、 decision-log.md 全都没进版本库。任何一次误操作（git checkout / 清理 / 换机器）都会抹掉两周的工作。 而且 git 历史现在完全无法讲述这段故事——最近的提交信息是「例外: 借用提交代码」×4。

建议动作：按主题切成 5-6 个提交（影子盘 / ST 诚实 / MC-1 口径 / 交易日历+tushare / 因子挖掘 / 数据断流修复）， 每个提交带上对应的 docs/feat/ 目录。这是下一铲的第一铲，做完之后所有其他重构才安全。

② 数据链的"诚实"是补丁式的，不是体系式的 07-15 断流三天 · 四道防线同时失守

昨晚的事故暴露了一个模式：每一层都会"假装成功"。离线桩 return [] 却 exit 0；空返回照样标履约水位； 数据门禁用全表 max(date)，一根指数 bar 就能掩护 5,200 只个股的陈旧；bat 不落日志所以没人知道。 已修的是这次的具体路径（refresh 强制在线 + fundamentals 履约诚实 + 三个 bat 落日志）， 但"沉默的降级"这个模式还在别处：SystemRiskGate 指数 bars<20 静默放行、熔断同步失败 fail-open、日亏闸取不到资产就跳过。

建议动作：数据门禁的新鲜度检查改成按 symbol 覆盖率（"5,198/5,200 只对齐今日"），而不是全表 max； 给所有 fail-open 分支加统一的"降级事件"通道，让降级本身成为可观测信号，而不是日志里的一行 warning。

③ 写了没接线的子系统 ≈ 2,000 行 组合风控 ~900行零引用 · 策略池 · 异常检测器 · DrawdownPolicy

这些代码质量不差、有测试、但生产链路里一次都没被实例化：
· 组合风控整棵子树（VaR / 压力测试 / 相关性 / 分散度 / ML 模型风险监控，~900 行）——只在自己模块内部互相引用，零外部装配
· 策略池治理（CANDIDATE→ACTIVE→RETIRED 状态机 + 评级引擎）——只被单测引用
· 异常检测器 ×4（策略/数据/市场/ML 模型）——聚合器 AnomalyDetector 存在，但没有任何组合根把检测器装进去
· DrawdownPolicy——全仓零引用，连测试都没有
· 死网关/死仓储：TushareIndexFetcher、telegram_gateway/wechat_gateway（另一套接口，factory 用的是 *_notifier）、 OrderRepository/SnapshotRepository + 它们依赖的孤儿 SQL 001_initial_schema.sql（现役交易走 TradingStore，根本不用这套表）

建议动作：照 REVIVAL.md 的老规矩办——删除清偿 + 记复活坐标。悬置两轮无人复活即答案。 每删一块，认知负荷就降一块；真需要时 git 历史里捞得回来。

④ 三套装配各写一份，宇宙/取数逻辑散在三处 _data_wiring · _backtest_wiring · _auto_trade_wiring

三个 wiring 模块各自组装 fetcher + store + registry + 宇宙解析。07-15 断流正是这个结构的产物—— 配置里一个 history_fetcher 字段，被"回测"和"补数"两种完全相反的语义共用，改一处炸另一处。 注释里还留着上一次同款事故的痕迹：三个回测入口曾各写一份，其中两个把宇宙随机截到 500 只（对微盘策略是静默的错误结论）。

建议动作：不急着大重构（会撞上未提交的工作区）。先做小的：把"取数意图"（在线补数 vs 离线读库）提升为 显式参数而非配置推断——force_online 已经开了这个头，继续把宇宙解析收敛成一个函数。

⑤ 双实现并存：有的是刻意，有的是遗留 对象式/向量化 ×4 对 · 手写指标 vs 特征引擎

刻意保留（非债）：factor_test 的 evaluator/runner/neutralizer/series 对象式与向量化双路——设计要求逐位等价， IR 计算与评分复用同一实现，是 --no-store 显式回退通道，golden 等价在案。
真遗留（是债）：CrossSectionBuilder._compute_bar_metrics（~125 行手写指标）与 feature_engine 并存， 且注释自陈手写版 return_20d 口径有偏（closes[0] bug，feature_engine 修了并定义 FEATURE_VERSION=1）。 主链路已走 feature_source，但这条错口径的路径还留着，随时可能被新调用方误用。
小瑕疵：MACD 柱因子在两个文件里算法完全相同只是名字不同；因子 .name 与 registry key 三处不一致（roe_quality vs roe 等）， 当前碰巧正确但脆弱；FactorScorer 同名不同物两个类。

建议动作：删 _compute_bar_metrics（把遗留调用方迁到 feature_source）；因子标识统一走 .name。

⑥ 文档失真与配置陷阱 codebase-map 停在 06-11 · user-guide 指向已删模块 · backtest.yaml 的 gitignore 悖论

· codebase-map.md 自称"与代码同步 2026-06-11"——已经落后一个多月，期间 domain 多了 factor_test 归位、ST、日历等
· user-guide.html（标注 v2026.06.01）仍有指向已删除模块 algo_strategies.twap_strategy 的示例代码， 实时看板还写着早已不存在的 ws://localhost:8000/ws/dashboard
· gitignore 悖论：resources/backtest.yaml 写在 .gitignore 里，但它早已被 git 跟踪—— git 对已入索引的文件无视忽略规则。所以"我以为本地改口径不会入库"是错觉，改动照样会被提交（07-12 那次切 DuckDB 就是这么进版本库的，也就间接埋了 07-15 的雷）
· 债务台账本身很扎实（9 项待清偿 / 16 项挂账观察 / 9 项未动工，每条带关闭凭证，维护纪律写在表尾）——这是全项目文档里最可信的一份，本图的债务数字全部取自它

建议动作：刷新 codebase-map；user-guide 删掉指向幽灵模块的两节；决定 backtest.yaml 到底跟不跟踪—— 要么 git rm --cached 让忽略生效（推荐，配合 .example 模板），要么从 .gitignore 里删掉这行，别再骑墙。

⑦ 特征线断供（新挂债 FD-1） 5,113 只 stock_features 停在 07-03 · 昨晚新发现

B2 特征校验的"成熟区"判定用库内首根 bar + 预热天数，但重算只装载 warmup_start 起的窗口—— 老股窗头的 rolling NaN 必然落在"成熟区"里 → 整只拒入不履约 → 每轮 refresh 全量重算重拒（浪费 + 日志刷屏）。 实测只有 82 只次新推进到了 07-15，5,113 只停在 07-03。
不慌：交易链不消费 stock_features（07-14 采样在特征停摆下正常运转，已实证），只有 ML/因子研究线受影响。 而且这道校验昨晚恰好立功——bars 断流期间它正确拦住了残缺特征入库。

建议动作：修判定窗口而非撤防线——成熟区改用"本轮装载窗首根 bar + 20 交易日"，或让 NaN 检查只覆盖本轮 upsert 的目标区间。

乱与不乱的分界线

干净的部分（别动，是资产）：

  • 决策核归一（回测=实盘同一份代码）
  • 三层交易防线 + 六道盘前闸（纯函数，单一事实源）
  • 留痕体系（H1 幂等预写、启动对账、审计 append-only）
  • 工程门禁五件套（ruff/pytest 1449/前端哈希/数据 9 检/AST 分层）

  • 数据履约账本 fetch_meta 的设计（问题出在"假履约"，不在设计）

乱的部分（按顺序清）：

  • 未提交的两周工作（最急，其他都建立在这之上）
  • 沉默降级的模式（数据/风控多处 fail-open）
  • ~2,000 行未接线子系统（删除清偿）
  • 三套装配的语义混用（07-15 事故根结构）
  • 手写指标遗留路径（口径已知有偏）
  • 文档失真 + gitignore 骑墙

一句话总结

这个系统的骨架是好的——决策核归一、防线分层、留痕完整、门禁自动化， 这些是很多个人量化项目做不到的。它现在给人"乱"的感觉，主要来自三件事： 两周的工作悬在工作区没提交、约 2,000 行代码写了没接线、以及数据链上那些"假装成功"的沉默降级。 这三件事都是可以在一周内清完的，而清完之后，剩下的就是老老实实攒够 10 个影子盘样本——那个没有捷径。
